Overall workflow for the ELK service developed @NIPNE-HH | DFCTI.

The current ELK platform

Currently, there are two main development pipelines which involve log analysis. Each pipeline is a fully fledged ELK stack that runs on elk.nipne.ro VM, ingesting logs from different sources, both with completely different log formats, then the parsed data are stored into separate indices. With Kibana, visualizing & interpreting the data from each index is straightforward.

Services

The elk.nipne.ro virtual machine has also other development projects that focus on numerical programming, web development or software development.

  • A live-server that supports Visual Studio Code through the browser. This is accessible via web through HTTPS: code-server.

  • NGINX reverse proxy for accessing Kibana UI through the browser.

  • A full-stack MEAN application developed with Angular, that uploads any log-file to MongoDB but also sends the files further to the logstash instance which runs on the VM. See project here.

  • An XROOTD environment that allows the user to develop new methods and implementations for the framework.

The two ELK stacks developed at DFCTI and running on the VM

Parsing logs from a SLURM workload manager. A logstash configuration pipeline that listens on incoming beats on port 5044.

The incoming logs have a special format that requires a grok pattern and some kv filters. Data is also aggregated, and additional fields are created based on the existing ones for providing more information.

Project

Logstash is configured to listen on the same port, but it is ingesting logs from a difference source: nova-compute-nodes (kubernetes cluster).

The logs are delivered through filebeat, which is pre-configured on the external cluster to create unique tags and identifiers for each log file (current pipeline supports file batch ingesting). Logstash then sends each file to the same ES. index.
